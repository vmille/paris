Le vrai problème : Aucune vérification formelle

Ce que la documentation dit sur std::sort
std::sort | Algorithm library

template < class RandomIt >
void sort ( RandomIt first, RandomIt last ) ;

template < class RandomIt, class Compare >
void sort ( RandomIt first, RandomIt last, Compare comp ) ;

Les vraies contraintes :
• RandomIt must be a random-access iterator
• The dereferenced type must be MoveAssignable
Les contraintes sont implicites et invérifiables

C++20 Concepts : La solution formelle

Les concepts formalisent
les contraintes sur les templates

Avant (C++17)

template<typename T>
void sort(T first, T last) {
  // T doit être random-access
  // Espérons-le !
}

Maintenant (C++20)

void sort(std::random_access_iterator auto first,
                           std::random_access_iterator auto last) {
  // Vérifié formellement
}

Vos concepts métier

Créez des contraintes spécifiques à votre domaine
// Concept personnalisé : Sérialisation JSON
template<typename T>
concept JsonSerializable = requires(T obj) {
  { obj.toJson() } -> std::convertible_to<std::string>;
};

void saveToFile(JsonSerializable auto obj, const std::string& filename) {
  file << obj.toJson();
}
Acceptés :

saveToFile(user, "user.json");
saveToFile(config, "cfg.json");
Rejetés :

saveToFile(myInt, "num.json");
error: int does not
satisfy JsonSerializable

L'unification : auto et templates ensemble

C++20 unifie les templates et auto

Syntaxe traditionnelle

template<typename T>
requires std::totally_ordered<T>
T findMax(T a, T b) {
  return (a > b) ? a : b;
}

Nouvelle syntaxe (C++20)

auto findMax(std::totally_ordered auto a,
          std::totally_ordered auto b) {
  return (a > b) ? a : b;
}
Plus concis, plus clair, plus sûr

Sous le capot : Comment ça marche

Structure d'un concept

template<typename T>
concept Comparable = requires(T a, T b) {
  { a < b } -> std::convertible_to<bool>;
  { a == b } -> std::convertible_to<bool>;
  { a != b } -> std::convertible_to<bool>;
};

// Type satisfait le concept SI ET SEULEMENT SI
// toutes les expressions compilent ET retournent bons types

Concept satisfait

int a = 5, b = 10;
a < b; // bool
a == b; // bool
a != b; // bool

Comparable<int> ✓

Concept non satisfait

struct Custom {};
Custom a, b;
a < b; // Non défini
a == b; // Non défini

Comparable<Custom> ✗

SFINAE + enable_if: L'ancien mécanisme

SFINAE : Substitution Failure Is Not An Error

Avant C++20, pour vérifier si un type a une méthode, on utilisait SFINAE avec enable_if. C'est complexe et fragile.

Approche SFINAE (C++17)

template<typename T>
typename std::enable_if_t<
  std::is_invocable_v<T>
>
execute(T f) {
  f();
}

// Lourd, peu lisible

Approche Concepts (C++20)

void execute(std::invocable auto f) {
  f();
}

// Clair, idiomatique
// Concepts remplacent
// SFINAE

Concepts + Overloading

// Distinction claire grâce aux concepts
void process(std::integral auto x) { }
void process(std::floating_point auto x) { }

process(42); // Version integral
process(3.14); // Version floating_point
